## Adding Content

Content lives as `.md` files in `contents/`. No TypeScript knowledge is needed — just create a Markdown file with YAML frontmatter.

### Supported Categories

| Folder | Category | Description |
|--------|----------|-------------|
| `contents/prompts/` | Prompts | Reusable prompt templates |
| `contents/skills/` | Skills | Skill guides and techniques |
| `contents/tips/` | Tips | Quick tips and best practices |
| `contents/docs/` | Docs | Documentation references |
| `contents/links/` | Links | Useful external resources |
| `contents/agents/` | Agents | Agent definitions and configs |

### File Format

Every content file must have YAML frontmatter at the top:

```markdown
---
name: Your Content Title
description: A concise one-line description
tags:
  - tag1
  - tag2
url: https://optional-link.com  # optional
---

Your full Markdown content goes here.

Supports **bold**, *italic*, `code`, lists, tables, and all GFM features.
```

### Frontmatter Fields

| Field | Required | Description |
|-------|----------|-------------|
| `name` | Yes | Display title for the card |
| `description` | Yes | Short description shown on the card |
| `tags` | Yes | Array of tags for filtering and search |
| `url` | No | External link (shown as a button in the modal) |

### File Naming

- Use lowercase kebab-case: `my-new-prompt.md`
- Agent files can use `.agent.md` suffix: `my-agent.agent.md`
- Avoid spaces and special characters

### Quick Example

To add a new prompt, create `contents/prompts/unit-test-generator.md`:

```markdown
---
name: Unit Test Generator
description: Generate comprehensive unit tests for any function
tags:
  - testing
  - automation
  - copilot
---

## Context

You are a test engineer. Generate unit tests for the provided function.

## Instructions

1. Cover happy path, edge cases, and error scenarios
2. Use the project's existing test framework
3. Follow AAA pattern (Arrange, Act, Assert)
4. Include descriptive test names
```

Then run `pnpm run build-content` or restart the dev server.

## Development Setup

### Prerequisites

- Node.js 18+
- pnpm (recommended)

### Getting Started

```bash
# Clone the repo
git clone https://github.com/Abhiram-MandS/ai-labs.git
cd ai-labs

# Install dependencies
pnpm install

# Start dev server (auto-runs build-content)
pnpm run dev
```

The app will be available at [http://localhost:4200](http://localhost:4200).

### Useful Commands

| Command | Description |
|---------|-------------|
| `pnpm run dev` | Start dev server with hot reload |
| `pnpm run build` | Production build |
| `pnpm run build-content` | Regenerate `lib/content-data.ts` from `.md` files |
| `pnpm run lint` | Run ESLint |

## Build Pipeline

```
build-content  →  next build
```

1. **`build-content`** — Reads all `.md` files from `contents/`, parses frontmatter with `gray-matter`, and generates `lib/content-data.ts`. This file is gitignored.

2. **`next build`** — Standard Next.js production build.

### Auto-generated Files (do not edit)

- `lib/content-data.ts` — Generated by `build-content`

This is listed in `.gitignore`.

## Pull Requests

### For Content Changes

1. Create a branch: `git checkout -b content/my-new-prompt`
2. Add your `.md` file to the appropriate `contents/` subfolder
3. Run `pnpm run build-content` to verify it compiles
4. Run `pnpm run dev` to preview
5. Commit and push
6. Open a PR with a clear title describing the content added

### For Code Changes

1. Create a branch: `git checkout -b feat/my-feature`
2. Make your changes
3. Run `pnpm run lint` to check for issues
4. Run `pnpm run build` to ensure a clean production build
5. Commit and push
6. Open a PR describing what changed and why

### Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add unit test generator prompt
fix: correct tag filtering for agents
docs: update README with new categories
chore: update dependencies
```

## Code Style

- **TypeScript** — Strict mode, no `any` where avoidable
- **Components** — Functional components with hooks
- **Styling** — Tailwind CSS utility classes (v4 syntax)
- **Naming** — PascalCase for components, camelCase for functions/variables, kebab-case for files
- **Imports** — Use `@/` path alias for project imports
